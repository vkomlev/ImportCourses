# Project Overrides - ParseCourse

## Цель
Фиксировать особенности парсера HTML-структуры курсов (сайт `victor-komlev.ru`) и экспорта в XLSX для импорта в LMS. Это не CMS и не работает с БД напрямую — своя БД у проекта отсутствует.

## Доменные ограничения
- Сохранять повторяемость и идемпотентность парсинга: один и тот же `--url` должен давать один и тот же набор `course_uid`/`external_uid` при неизменной странице.
- Явно разделять этапы: HTTP-загрузка страницы (`parser`) → извлечение сущностей (курсы, материалы) → генерация UID (`uid_generator`) → запись в XLSX (`excel_writer`).
- Изменения форматов колонок XLSX документировать в `docs/courses-import-manual.md` / `docs/materials-import-template.md` / `docs/materials-api.md` синхронно с изменением `src/excel_writer.py`.
- Селекторы разметки (заголовки, текст ссылок) держать в `src/config.py`, не хардкодить в `src/parser.py` — сайт-источник может менять вёрстку.

## Ограничения по внешним интеграциям
- Единственная внешняя зависимость — HTTP-доступ к сайту-источнику; менять URL/домен источника только при явной задаче.
- Импорт в LMS (Google Sheets + API) — вне кода этого проекта; менять контракт колонок только синхронно с LMS-стороной.
- `docs/ai/WORKFLOWS/db-change.md` для этого проекта неприменим (нет БД) — не использовать как чек-лист для изменений формата XLSX.

## Обязательные проверки перед merge в main
- Прогнать `pytest tests/ -v` (тесты на HTML-фикстурах, без сети).
- При изменении логики парсинга — smoke-прогон на реальном `--url` с `--verbose`.
- Проверить отсутствие дубликатов `course_uid`/`external_uid` в выходных XLSX.
- Проверить корректность логирования ошибок и итогового статуса (exit code).
- Получить `review-gate: PASS`.

## Ограничения по безопасностным действиям
- Не хранить учётные данные внешних сервисов (Service Account JSON, API-ключи LMS) в коде, логах или коммитах — только пути к секретам вне репозитория.
- Не выполнять массовые перезаписи `output/` или чужих XLSX без явного запроса.

## Исключения и особые кейсы
- Для исследовательских изменений (например, проверка новой структуры страницы) допустим ограниченный тестовый прогон на одном `--url`, но с обязательной фиксацией рисков (что могло сломаться при смене селекторов) в задаче/PR.
